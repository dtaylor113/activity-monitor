# Activity Monitor

Consolidated status report of recent GitHub PR activity and Jira ticket changes.

## Prerequisites

- `gh` CLI authenticated with `repo` and `read:org` scopes
- Environment variables set via `ocmui-tokens.sh` (see `setup-tokens.example.sh`):
  - `$GITHUB_USER` — GitHub username
  - `$GITHUB_REPO` — GitHub repo (org/repo format)
  - `$JIRA_EMAIL` — Atlassian email
  - `$JIRA_TOKEN` — Jira API token
  - `$JIRA_INSTANCE` — Jira hostname (e.g., `redhat.atlassian.net`)
  - `$JIRA_PROJECT` — Jira project key (e.g., `OCMUI`)

## State File

The skill maintains a persistent state file at:

```
ACTIVITY_MONITOR.md    (in the activity-monitor/ directory)
```

This file stores:
- `last_checked` timestamp (used as the SINCE boundary for the next run)
- The most recent structured summary in markdown (fallback view)

### Reading last_checked

Before gathering data, read `ACTIVITY_MONITOR.md` and extract the
`last_checked` value from the YAML frontmatter. Pass it to the script:

```bash
bash scripts/gather.sh --since "2026-05-21T15:23:00"
```

If the file doesn't exist or has no `last_checked`, fall back to 3-day lookback.

### Writing the files

After presenting the summary to the user, write these output files:

1. **`ACTIVITY_MONITOR.md`** — overwrite with the updated markdown report
   (same template as before, includes YAML frontmatter with `last_checked`).
2. **`activity-data.js`** — overwrite with the structured data (see below).
   This is the **ONLY file that changes on each skill run**.
3. **`ACTIVITY_MONITOR.html`** — static template that loads `activity-data.js`
   and renders the UI client-side. **DO NOT rewrite this file** unless adding
   new features/sections. It never changes for data-only updates.
   **IMPORTANT**: When you DO change the HTML (new features, layout, CSS, JS),
   you MUST also update this SKILL.md to reflect those changes (data format,
   HTML structure examples, feature descriptions, CSS classes, etc.).

The HTML loads `activity-data.js` via `<script src="activity-data.js">` and
renders everything dynamically. The user opens the HTML in a browser.

## Workflow

Run the helper script to gather raw data, then present a structured summary.

### 1. Gather data

```bash
# With last_checked from state file:
bash ~/.cursor/skills/activity-monitor/scripts/gather.sh --since "LAST_CHECKED_ISO"

# First run / fallback:
bash ~/.cursor/skills/activity-monitor/scripts/gather.sh 3
```

The script outputs JSON sections for each data source:
- `ALL_EPICS` — full list of active epics with current fields + last 3 comments each
- `PARENT_EPIC_STATUS` — parent alignment info and parent comments for all epics with parents
- `EPIC_CHILDREN` — open child stories per epic
- `CHILD_PR_STATUS` — GitHub PR info for children in Code Review/Review
  (matches PRs by searching for the Jira key in PR title first, then PR body/description)
- `SIBLINGS` — open children of each epic's parent (non-OCMUI tickets: backend, docs, etc.)
- `REVIEW_REQUESTS`, `STALE_REVIEWS`, `PR_COMMENTS` — PR data

The `ALL_EPICS` section provides the complete epic list every run (no need to
carry forward from previous runs). Map it directly to the `epics` array in
`activity-data.js`. **Important:** `ALL_EPICS` does not include `parent_target_end`
— after assembly, merge `parent_target_end` from `PARENT_EPIC_STATUS` into each
epic's entry (the parent alignment data has the correct value).

**AI-generated content (written by the agent, not Python keyword extraction):**

After assembling raw data from gather.sh, the agent reads all comments and
writes synthesized AI summaries. These are NOT direct quotes or vague topic
labels — they describe the substance of discussions at a useful level of detail.

Good: "Bug: Terraform provider fails when enabling AutoNode. Team has fix ready, need PM input on severity."
Good: "You commented yesterday, awaiting reply from jmekkatt."
Good: "4 Dev + 3 QE + 3 Flag open. 8 unassigned. All still To Do — not yet started"
Bad (too vague): "questions raised, bug discussed"
Bad (too vague): "Network tags + E2E." — says nothing about status or blockers
Bad (too vague): "3 stories. PR #454 fixed." — which stories? what's their status?
Bad (too literal): "They have a PR fix they can contribute, but t..."

**"Ball in court" rule:** For PR AI summaries, always note who has the ball:
- If $GITHUB_USER commented last → "You commented last (date), awaiting reply from X."
- If someone else commented last → "Last comment from X (date) — may need your response."
This tells the user at a glance whether they need to act or are waiting.

**Age indicators:** Every AI summary must be prefixed with a freshness label based
on the most recent activity date being summarized:
- `@ today`, `@ 1 day ago`, `@ N days ago` (under 14 days)
- `@ ~N weeks ago` (14–59 days)
- `@ ~N months ago` (60+ days)

This tells the reader whether the discussion is active or stale.

For **children summaries**, the age uses ALL activity sources — not just Jira comments:
- Jira comments on child tickets
- Child PR comments (from `child_pr_status`)
- Child PR updated dates

The age label includes the **source** in parentheses: `@ 3 days ago (PR)` or `@ today (Jira)`.
This tells the reader where the most recent activity is happening.

**Fields that receive AI summaries:**
- `epics[].comments_ai` — synthesis of the epic's latest comments
- `epics[].uber_ai` — **uber summary** that synthesizes ALL sub-sections (comments,
  children, child PRs, parent, siblings) into one high-level assessment. Highlights
  urgent/time-sensitive issues with indicators:
  - 🔴 URGENT — active blockers requiring immediate action
  - ⚠️ TARGET PAST DUE — missed deadlines or stalled work
  - 🟡 — things moving but need attention
  - 🟢 READY — positive status, waiting on external gate
  - Plain text — low urgency / backlog
- `parent_comments_ai.{KEY}` — synthesis of parent ticket comments
- `epic_children.{KEY}.ai_summary` — children status with **category breakdown** and per-status PR detail:
  - Categories: **Dev** (default), **QE** ("E2E Automation"/"Post-merge testing"/"CI Automation" in title), **Flag** ("Feature Flag" in title)
  - Format: `"N Cat (X To Do, Y In Progress, Z Code Review: #PR review-status, W Review: #PR review-status)"`
  - For Code Review/Review statuses, include the PR number and reviewer breakdown
  - Join categories with ` + `
  - Review status uses: `N approved`, `N changes requested`, `N pending`, or `no reviewers yet`
  - Example: `"@ 3 days ago (Jira) — 5 Dev (3 To Do, 1 Code Review: #515 no reviewers yet, 1 Review: #454 2 approved, 1 changes requested) + 2 QE (1 To Do, 1 In Progress) + 2 Flag (2 To Do)"`
- `siblings_ai.{KEY}` — auto-generated summary grouped by Jira ticket prefix with
  status counts per prefix. Format: `"@ <age> (<source>) — PREFIX1 (N Status1, N Status2) + PREFIX2 (N Status3)"`.
  Example: `"@ 1 day ago (Jira) — ROSAENG (1 To Do, 2 In Progress) + OSDOCS (1 In Progress) + PERFSCALE (1 Release Pending)"`
- `prs[].ai_summary` — PR-level summary: staleness, comment themes, blocking issues
- `child_pr_status.{KEY}.ai_summary` — child PR comment synthesis

If the script isn't available or fails, fall back to the manual queries below.

### Expandable Epic Detail Rows

Each epic row in the Active Epics table has a **chevron toggle** (`▶`) in the
Key column. Clicking it expands a hidden detail row below that epic, spanning
all columns. The detail row contains a panel starting with an **uber AI summary**
(cross-section synthesis with urgency indicators), then an **"Expand All" button**
that toggles visibility of section details. Each sub-section shows its title +
AI summary by default, with raw data (comments, child lists, etc.) in a
**collapsed `.section-collapsible` div** that only shows when expanded.
Sub-sections:

1. **Latest Comments**: Last 8 human comments on the epic itself (author,
   date, body snippet). Only shown if the epic has comments.
2. **Children (N open)**: Summary count + list of open child stories/tasks.
   Each child shows key (linked), summary, status, assignee, and latest
   comment (author, date, body snippet). Only shown if the epic has open
   children. Data comes from `EPIC_CHILDREN` section of gather.sh.
   **PR Status for Code Review children**: For children in "Code Review" or
   "Review" status, a PR info panel is shown below the child entry with:
   - GitHub PR number (linked), author
   - Approval count (green ✓N) and requested changes count (red ✗N)
   - Last 8 unresolved non-bot comments (user, date, body snippet)
   Data comes from `CHILD_PR_STATUS` section of gather.sh.
3. **Parent Alignment**: Parent key (linked), summary, target end, and
   "DATE MISMATCH" flag if dates differ. Only shown if the epic has a parent.
4. **Parent Comments**: Last 8 human comments from the parent ticket. Only
   shown if the epic is within 10 days of target end or past due AND has a
   parent with recent comments.

**Sort/filter behavior**:
- Sorting keeps `tr.epic-detail` rows paired with their parent epic row
- Filtering hides both the epic row and its detail row when filtered out
- If a detail row is expanded and then the user sorts, it stays expanded

**Target End color rules** (HTML: CSS class `.red` or `.orange`):
- **Red** (`.red`): Target End is past due OR Target End > Parent Target End (OCMUI is late vs feature)
- **Orange** (`.orange`): within 10 days (not past due, not late vs parent)
- **Plain**: more than 10 days away and not late vs parent

**Parent Target End color rules**:
- **Red** (`.red`): OCMUI Target End > Parent Target End (UI deadline is after the feature deadline)
- **Plain**: OCMUI is on time or early relative to parent (including dates that don't match but OCMUI is earlier)
- **Plain**: matches or one/both are unset

**PRs Needing My Attention** must include:
- Review requests assigned to me (`REVIEW_REQUESTS` section)
- PRs where my review is stale — reviewed but new commits pushed (`STALE_REVIEWS` section)
- PRs I authored with new review comments (`PR_COMMENTS` section)
- For each PR, fetch recent human comments (filter out bots: codecov, coderabbitai)
  and display as sub-rows beneath the PR

**Epic detail row — Latest Comments** shows:
- The 3 most recent human comments on the epic (bot comments filtered out)
- Each entry: author name, created date, body snippet (up to 200 chars)
- Comments come directly from the `ALL_EPICS` section of gather.sh

**Epic detail row — Parent Alignment** flags:
- Date mismatches (OCMUI target ≠ parent target) — may indicate stale dates
- Parent target end changes — OCMUI epic may need to adjust
- Parent status changes (e.g., Refinement → Analysis)

**Epic detail row — Parent Comments**:
When an OCMUI epic is **within 10 days of its target end** (or past due), the
script fetches the last 3 comments from the parent feature. Surface these in
the expandable detail row with author, date, and body.

### Linking Conventions

All Jira IDs and PR numbers in the output MUST be markdown links:
- Jira: `[OCMUI-XXXX](https://${JIRA_INSTANCE}/browse/OCMUI-XXXX)`
- GitHub PR: `[#NNN](https://github.com/${GITHUB_REPO}/pull/NNN)`

### CRITICAL: Always Include Summary Context

**Every single Jira ID** that appears anywhere in the output — tables, bullet
points, inline references — MUST be accompanied by a short summary so the
reader never has to click through to understand what a ticket is about.

Formats:
- In a table with a Summary column: the column provides the context
- In the Parent column: `[ROSA-377](url) (Fetch updates from CVO)`
- In bullet points: `**[OCMUI-4020](url)** (Y-stream channel selection): ...`
- Inline: `...parent [ROSA-17](url) (AutoNode GA) shifted...`

**NEVER** output a bare Jira key like `ROSA-377` or `OCMUI-3580` without
a parenthetical summary. This applies to ALL projects (OCMUI, ROSA, OSDGCP,
OCPSTRAT, etc.).

## Manual Queries (Fallback)

If the script is unavailable, run these directly:

### Active Epics (all statuses)

```bash
curl -s -u "$JIRA_EMAIL:$JIRA_TOKEN" \
  "https://${JIRA_INSTANCE}/rest/api/3/search/jql" \
  -G \
  --data-urlencode "jql=project = ${JIRA_PROJECT} AND issuetype = Epic AND status in (\"In Progress\", Review, Refinement, Backlog) ORDER BY \"Target end\" ASC, priority DESC" \
  --data-urlencode 'maxResults=30' \
  --data-urlencode 'fields=key,summary,status,assignee,priority,updated,customfield_10023,customfield_10542' \
  --data-urlencode 'expand=changelog'
```

### Jira Custom Fields Reference

| Field ID | Label |
|----------|-------|
| `customfield_10023` | Target end |
| `customfield_10542` | Marketing Impact Notes |
| `customfield_10483` | Blocked Reason |

### PRs Involving Me (updated in last 3 days)

```bash
gh api search/issues \
  -F "q=repo:${GITHUB_REPO} is:pr involves:${GITHUB_USER} updated:>$(date -v-3d +%Y-%m-%d)" \
  -F per_page=20 \
  --jq '.items[] | {number, title, updated_at, state}'
```

### Recent review comments on my PRs

```bash
gh api repos/${GITHUB_REPO}/pulls/{number}/comments \
  --jq '[.[] | select(.updated_at > "SINCE_DATE") | {user: .user.login, body: .body[:100], updated_at}]'
```

### PR reviews assigned to me

```bash
gh api search/issues \
  -F "q=repo:${GITHUB_REPO} is:pr is:open review-requested:${GITHUB_USER}" \
  -F per_page=10 \
  --jq '.items[] | {number, title, author: .user.login}'
```

### PRs where my review is stale (reviewed but new commits pushed)

```bash
gh api search/issues \
  -F "q=repo:${GITHUB_REPO} is:pr is:open reviewed-by:${GITHUB_USER} -review:approved -author:${GITHUB_USER}" \
  -F per_page=10 \
  --jq '.items[] | {number, title, author: .user.login, updated_at}'
```

### Current sprint items

```bash
curl -s -u "$JIRA_EMAIL:$JIRA_TOKEN" \
  "https://${JIRA_INSTANCE}/rest/api/3/search/jql" \
  -G \
  --data-urlencode "jql=project = ${JIRA_PROJECT} AND assignee = currentUser() AND sprint in openSprints() ORDER BY status ASC" \
  --data-urlencode 'maxResults=30' \
  --data-urlencode 'fields=key,summary,status,issuetype,priority,labels,customfield_10020'
```

### HTML Table Columns

**Active Epics**: Key | Summary | Assignee | Status | Target End | Parent Target End | Marketing Impact Notes | Parent
- Key column includes the expand toggle chevron before the Jira link
- Expandable detail rows (see "Expandable Epic Detail Rows" above) replace the
  standalone "Epic Status Changes", "Parent Feature Alignment", and "Latest
  Parent Comments" sections. Those sections NO LONGER exist as separate tables.

**GitHub PRs** (split into three tables):

1. **My PRs** — PRs authored by `$GITHUB_USER`
2. **PRs I'm Reviewing** — PRs where the user is a requested reviewer (individually
   or via team membership) and has NOT yet approved. States shown: pending, commented,
   changes_requested. Sorted by the user's own review state priority:
   pending (awaiting review) → commented (left a review comment) → changes_requested
   (already blocked). Within same priority, sorted by oldest updated first.
3. **PRs I've Approved** — PRs where the user's review state is `approved`. Separated
   to reduce noise since these no longer require action.

The filter uses `pr_status[N].reviewers` to determine membership. Team-based review
requests are resolved by fetching the user's GitHub teams (`user/teams` endpoint) and
adding the user as "pending" when a requested team matches.

Columns: PR | Title | Author | Reviewers | Checks | AI Summary | Updated
- Title = full PR title (includes Jira ticket ID prefix, e.g. "OCMUI-4330: ...")
- Author = GitHub username who opened the PR
- Reviewers = individual reviewers with colored status (approved/changes_requested/commented/pending)
- Checks = CI status (passing/failing/pending) + merge status (mergeable/blocked/conflicts/unstable)
- AI Summary = synthesized comment themes and staleness (`.ai-text` styled)
- Updated = date of last activity

  current PAT lacks. This skill works around it by querying PR/review endpoints
  directly.
